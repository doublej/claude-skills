CORE FINDINGS:

Problem 1: MCP Coverage Is Insufficient
────────────────────────────────────────
Shopify API: ~1000 operations
GeLi2001/shopify-mcp: 9 tools (0.9% coverage)
amir-bengherbi/shopify-mcp: 15 tools (1.5% coverage)

Users hit the 98.5% gap immediately on real workflows.
MCP creates false expectations of completeness.

Problem 2: Quarterly Versioning = Maintenance Hell
──────────────────────────────────────────────────
Shopify releases 4 new API versions per year.
Each version requires: testing, updates, deprecation handling.
3-year effort: 40+ hours/year (growing).

Existing MCPs show no version management track record.

Problem 3: GraphQL Cost-Based Rate Limiting
──────────────────────────────────────────
3,000,000 points/day budget (resets at midnight UTC)
Easy to burn budget silently (no built-in MCP monitoring)
Users blame tool: "Why did my integration stop?"

Problem 4: Credential Storage Risk
──────────────────────────────────
MCP holds Shopify access tokens in memory.
Single point of compromise = all connected stores at risk.
No built-in token rotation/revocation patterns.

Problem 5: Architectural Mismatch
─────────────────────────────────
Skills are documentation + patterns, not stateful proxies.
Real Shopify workflows require: webhooks, background jobs, bulk ops.
MCP request-response model inadequate for these patterns.

SOLUTION: "Shopify API Patterns" Skill

Why This Works:
✓ 100% coverage via official references
✓ Low maintenance: ~2 hours/quarter (mostly link checks)
✓ Teaches rate-limit awareness (not a footgun)
✓ No credential storage
✓ Sustainable 3-year outlook
✓ Empowers users to build with official tools

Build Effort:
Initial: 5-7 days (Phase 1-4)
Quarterly: 1-2 hours maintenance
3-year total: ~10-12 hours additional maintenance
